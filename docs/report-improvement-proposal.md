# 自己分析レポート改善まとめ

## 課題と対応一覧

| 課題 | 問題だったこと | 対応 |
|---|---|---|
| 出力が抽象的 | 「達成率が高かった」程度の一般論しか出ない | 施策1: 出力例をプロンプトに埋め込む |
| 新規ユーザーへの対応 | タスクが少ないと「データ不足」で終わる | 施策2: データ数に応じてプロンプトを切り替える |
| 欠損データの無視 | 未記入の達成率・振り返りをLLMがスルーする | 施策3: 欠損状況を先頭にサマリーとして渡す |

---

## 施策1: 出力例の埋め込み

### やったこと

2つの技法を組み合わせて、LLMが具体的な分析を出力するよう誘導します。

**1a. Exemplar Injection（参照データ注入）**

ユーザーの実タスクから「達成率100% かつ タイムブロッキング済み」のタスクを最大3件抽出し、コンテキスト末尾に追加します。

```
## 成功事例（参照データ）
以下は達成率100%を記録したタスクの実例です。これと同様の粒度で分析してください。
- **競合調査レポート作成**（開始: 09:00〜11:00、自己見積もり: 2.0h、優先度: 高）→ 達成率100%
```

→ 実データを見せることで「このくらい具体的に書いて」と伝える

**1b. True Few-shot Prompting**

架空のユーザーデータ → 期待する分析文のペアをプロンプト先頭に埋め込みます。

```
### 参考例1（過負荷傾向のユーザー）
入力: 合計18件 / 完了7件、自己見積もり1h → AI見積もり3.5h...
出力: 今週は18件中7件完了（達成率39%）。AI見積もりとの乖離が2h以上の...
```

→ 「こういうフォーマット・粒度で書いて」とスタイルを学習させる

### 2つの違い

| | 1a: Exemplar Injection | 1b: True Few-shot |
|---|---|---|
| データ | ユーザーの実タスク | 架空の合成データ |
| 場所 | コンテキスト末尾（Go側で動的生成） | プロンプト先頭（Python側で固定） |
| 目的 | 分析の根拠を提示する | 出力スタイルを学習させる |

### 変更ファイル

- `backend/internal/service/context_builder.go` — `buildFewShotSection()` で実タスクを抽出・整形
- `ai/app/prompts/report.py` — `_FEW_SHOT_EXAMPLES` を定数として定義し `REPORT_PROMPT` 先頭に結合

## 施策2: コールドスタート対応

### やったこと

タスクが5件未満のユーザーには、データドリブン分析ではなく行動経済学ベースの仮説提示に切り替えます。

```
タスク件数 < 5件 → COLD_START_PROMPT（仮説型）
タスク件数 ≥ 5件 → REPORT_PROMPT（分析型）
```

### なぜ5件が閾値か

5件未満では成功・失敗パターンを統計的に語れないため、「あなたのデータではこう見えます」ではなく「こういう傾向がある人が多いです、該当しそうですか？」という仮説提示の方が価値がある。

### COLD_START_PROMPT の構成

1. **現状サマリー** — 件数・完了状況を事実ベースで記述
2. **行動傾向の仮説** — 計画錯誤・現在バイアス・コンフォートゾーンと照合して2〜3個の仮説を提示
3. **今すぐ試せる改善行動** — 1つだけ具体的に提示
4. **データ蓄積のすすめ** — 次回レポートの精度を上げるための記録アドバイス

### 変更ファイル

- `ai/app/services/report_service.py` — `_select_prompt()` でコンテキストの「合計: N件」を正規表現で抽出して切り替え
- `ai/app/prompts/report.py` — `COLD_START_PROMPT` を新規追加

## 施策3: データ欠損の明示

### やったこと

コンテキストの先頭に「データ品質サマリー」を追加し、LLMが欠損を認識した上で分析できるようにします。

```
## データ品質サマリー
- 振り返り記録あり: 3日 / 記録なし: 1日
- 達成率入力済み: 8タスク / 未入力: 5タスク
- タイムブロッキング済み: 6タスク / 未配置: 7タスク
```

### なぜこれが必要か

LLMは「（記録なし）」というテキストをそのままスルーしがち。欠損の件数を数値で先頭に置くことで、「このデータには信頼性の限界がある」と認識させることができる。

### 変更ファイル

- `backend/internal/service/context_builder.go` — `buildDataQualitySummary()` で欠損件数を集計・整形

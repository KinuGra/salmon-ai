# 自己分析レポート改善 提案書

## 背景

現在の `REPORT_PROMPT` はシンプルな指示文のみで、以下の3つの問題を抱えている。

| 問題 | 現状 | 影響 |
|---|---|---|
| Few-shot なし | LLM に分析の粒度を示す例がない | 「成功パターン：達成率が高かった」程度の抽象的な記述で終わる |
| コールドスタート未対応 | データが少なくても同じプロンプトを使う | 「データが不足しています」という空虚なレポートになる |
| データ欠損の無視 | 振り返り未記入・達成率 null を `（記録なし）` と書くだけ | LLM がノイズとして無視し、欠損を指摘しない |

---

## 施策1: Exemplar Injection（参照データ注入）+ True Few-shot Prompting

### 用語の整理

本施策では2つの技法を組み合わせて実装しています。

| 技法 | 正式名称 | 内容 |
|---|---|---|
| 施策1a | **Exemplar Injection（参照データ注入）** | ユーザーの実タスクデータ（達成率100%）をコンテキストに挿入し、同粒度の分析を促す |
| 施策1b | **True Few-shot Prompting**（Brown et al., 2020） | 合成した「入力→出力」ペアをプロンプト内に埋め込み、LLMに分析スタイルを学習させる |

> **注意**: 「成功タスクをコンテキストに追加する」だけでは Few-shot Prompting ではなく Exemplar Injection です。
> True Few-shot Prompting は「入力例 → 期待される出力例」のペアを示すことで、LLM が出力フォーマットを模倣します。

### 理論的根拠

LLM の Few-shot Prompting（Brown et al., 2020）は、期待する出力の形式・粒度・視点を「入力→出力ペア」で例示することで、抽象的な指示より具体的で一貫した出力を引き出す手法。

> 「成功パターンを教えてください」より「**例のような形式で**成功パターンを教えてください」の方が、LLM の出力は具体的になる。

特に本アプリのユースケース（タスク×時間×達成率の相関分析）では、「15分の散歩後にDeep Workを配置すると達成率が20%上がる」のような **数値と行動が結びついた例** を示すことで、LLM が同様の粒度で分析するよう誘導できる。

### 現状の問題

`REPORT_PROMPT` には `{context}` だけが渡され、出力の粒度を示す例がない。
LLM は「達成率が高いタスクに成功パターンが見られます」という一般論で返しやすい。

### 実装方針

**施策1a: Exemplar Injection — Go 側（`context_builder.go`）で実タスクを自動抽出してコンテキストに含める。**

`BuildFullContext` に `buildFewShotSection` を追加し、以下の条件でタスクを最大3件抽出する：
- `achievement_rate = 100`
- `start_time IS NOT NULL`（タイムブロッキングされている）
- `estimated_hours IS NOT NULL`

抽出したタスクを「成功事例（参照データ）」セクションとしてコンテキスト末尾に追加する。

**施策1b: True Few-shot — Python 側（`report.py`）でプロンプト先頭に合成入力→出力ペアを埋め込む。**

```python
_FEW_SHOT_EXAMPLES = """
## Few-shot 参考例（出力スタイルの模倣用）

### 参考例1（過負荷傾向のユーザー）
入力データ（抜粋）: 合計18件 / 完了7件、自己見積もり1h → AI見積もり3.5h...
期待される分析文: 今週は18件中7件完了（達成率39%）。AI見積もりとの乖離が2h以上の...
"""

REPORT_PROMPT = _FEW_SHOT_EXAMPLES + "...{context}..."
```

### トレードオフ

| メリット | デメリット |
|---|---|
| 出力の具体性・再現性が上がる | コンテキスト長が増える（少量なので許容範囲） |
| 実データ（1a）と合成例（1b）の両方で誘導できる | 合成例が実ユーザーと乖離する可能性（→ 汎用的な例を選定） |
| 実データに基づくため説得力がある | 成功例がゼロの場合、1aは空になる（→ 条件分岐で対処） |

## 施策2: コールドスタート問題の解決

### 理論的根拠

行動経済学の**「計画錯誤（Planning Fallacy）」**（Kahneman & Tversky, 1979）：
人は自分の将来の行動について楽観的に見積もる傾向があり、特にデータが少ないうちは「自分は平均より上」というバイアスが強い。

データが少ない新規ユーザーへの対処として有効なアプローチ：
1. **Inside View → Outside View の転換**：個人データが少ない間は「同種のタスクを持つ一般的なユーザーの傾向（Outside View）」を先に示し、徐々に個人データ（Inside View）に移行する
2. **仮説提示型コーチング**：「あなたのデータからは○○の傾向が見られる可能性があります。今後のデータで検証しましょう」という書き方で、少データでも価値あるフィードバックを提供する

### 閾値の定義

| タスク件数 | 判定 | 対応プロンプト |
|---|---|---|
| 0〜4件 | コールドスタート | `COLD_START_PROMPT`（行動経済学ベースの仮説提示） |
| 5件以上 | 通常 | `REPORT_PROMPT`（データドリブン分析） |

タスク4件以下を閾値とする根拠：5つ未満のタスクでは、成功・失敗パターンの統計的有意性が低く、LLM の分析が過学習（少数事例の過大評価）になりやすいため。

### 実装方針

**Python 側（`report_service.py`）でコンテキスト文字列からタスク件数を抽出して判定。**

```python
# contextの "合計: N件" から件数を抽出
import re
match = re.search(r'合計:\s*(\d+)件', req.context)
task_count = int(match.group(1)) if match else 0

prompt_template = COLD_START_PROMPT if task_count < 5 else REPORT_PROMPT
```

**`COLD_START_PROMPT` の設計方針：**

- Role: 行動科学の知見を持つコーチ
- 一般的な傾向（計画錯誤・コンフォートゾーン・エネルギー管理）を先に提示
- ユーザーの少ないデータと照合して「仮説」として提示
- 「データが蓄積されるにつれてより精度が上がります」と次回への動機付けを含める

### トレードオフ

| メリット | デメリット |
|---|---|
| 新規ユーザーに価値あるフィードバックを提供できる | 件数判定がテキストパースに依存（context_builder の出力形式変更に弱い） |
| 離脱防止になる（空っぽのレポートで諦めない） | プロンプトが2種類になり保守コストが増える |

## 施策3: データクレンジング

### 理論的根拠

LLM は入力データの欠損を「明示的に指示しない限りスルーする」傾向がある。
`（記録なし）` というテキストは LLM にとって「内容なし」として処理され、「なぜ記録がないのか」「欠損がどう分析に影響するか」という洞察は生まれない。

**Missing Data の処理方針（統計的観点）：**

| 欠損種別 | 解釈 | プロンプト上の処理 |
|---|---|---|
| 振り返り未記入 | その日に内省できなかった可能性 | 欠損件数と日付をサマリーとして先頭に明記 |
| 達成率 null | タイムブロッキングされていないか入力漏れ | 未入力タスク数をサマリーに明記・分析から除外 |
| 実作業時間なし | start/end 未設定 → タイムブロッキング未活用 | 未使用の割合をサマリーに明記 |

### 実装方針

**Go 側（`context_builder.go`）にデータ品質サマリー生成を追加。**

`BuildFullContext` の先頭にデータ品質サマリーセクションを追加：

```
## データ品質サマリー

- 振り返り記録あり: N日 / 記録なし: M日
- 達成率入力済み: Nタスク / 未入力: Mタスク
- タイムブロッキング済み: Nタスク / 未配置: Mタスク
- 注意: 未入力データは分析から除外してください
```

**プロンプト側への追記（`REPORT_PROMPT`）：**

```
## データ品質への注意
- 達成率が未入力のタスクは分析対象から除外し、その旨をレポートに明記する
- 振り返りが記録されていない日は「欠損」として扱い、継続的な記録の重要性を指摘する
- データ欠損が多い場合は、分析の信頼性に限界があることを注記する
```

### トレードオフ

| メリット | デメリット |
|---|---|
| LLM が欠損を認識した上で分析する | コンテキスト長が増える |
| ユーザーに「入力漏れがある」と気づかせる教育効果がある | 欠損が多いと分析対象が少なくなりレポートが薄くなる（実態を反映しているので許容） |

## 実装しない判断をしたこと

### JSON 出力への変更
提案書ドラフトで「JSON 出力の検討」が挙げられていたが、今回は見送る。

理由：
- フロントエンドの `MarkdownRenderer` が既存で動作している
- Markdown → JSON への変更はフロントエンドの再実装コストが高い
- LLM の JSON 出力は Markdown より出力安定性が低く、パース失敗リスクがある

### 相関係数の事前計算
「振り返りに出現する単語と達成率の相関係数を計算してPythonに渡す」案は見送る。

理由：
- 現状の振り返りデータ量（1〜4件）では統計的有意性がない
- データが蓄積した後の改善候補として保留
